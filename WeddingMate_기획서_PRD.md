# Part 1. 서비스 기획서

---

## 1. Executive Summary

**웨딩메이트(WeddingMate)**는 결혼 준비부터 신혼 생활까지의 전 과정을 하나의 플랫폼에서 지원하는 올인원 라이프스타일 서비스입니다.

기존 한국 웨딩 앱 시장(연 20조 원 규모)은 "결혼 준비"에만 집중되어 있으며, 결혼 이후의 신혼 생활(집, 임신, 육아, 가족 행사 관리)을 통합 지원하는 서비스는 사실상 부재합니다. 웨딩메이트는 이 갭을 메우며, **결혼 준비 → 신혼 → 육아 초기**까지 이어지는 라이프사이클 전체를 커버하는 최초의 통합 플랫폼을 지향합니다.

**핵심 차별점:**
- 결혼 준비 + 신혼 생활 통합 관리 (기존 경쟁사 대비 유일)
- 지역 기반 SNS 커뮤니티 (인스타그램형 피드)
- 웨딩/신혼 특화 중고거래 마켓플레이스
- B2B 업체 입점 마켓플레이스 (예물, 스드메, 가전 등)
- 커플 공동 관리 시스템 (일정, 예산, 체크리스트)

**비즈니스 모델:** 커머스 수수료 + 프리미엄 구독 + 광고 + 중고거래 수수료의 복합 모델

## 2. 프로젝트 배경 및 목적

### 2.1 배경

한국의 결혼 시장은 연간 약 20조 원 규모이며, 매년 약 19만~20만 쌍이 결혼합니다(2024년 기준). 결혼 준비 과정은 평균 6~12개월이 소요되며, 결혼 비용의 불투명성, 정보 비대칭, 복잡한 의사결정이 예비부부들의 가장 큰 고충입니다.

더불어 결혼 이후 신혼부부가 직면하는 과제들(신혼집 꾸미기, 임신/출산 준비, 육아, 양가 가족 행사 관리 등)은 각기 다른 앱과 서비스에 분산되어 있어, 하나의 플랫폼에서 통합 관리할 수 없는 상황입니다.

### 2.2 문제 정의

| 구분 | 문제점 | 현재 해결 방식 | 한계 |
|------|--------|---------------|------|
| 결혼 준비 | 예물/스드메/웨딩홀 가격 불투명 | 웨딩북, 오딩 등 개별 앱 | 결혼 이후 사용 중단 |
| 결혼 준비 | 일정/예산 관리 복잡 | 엑셀, 노션 | 커플 공유 불편 |
| 신혼 생활 | 신혼집 구하기/인테리어 | 직방, 오늘의집 등 분산 | 신혼 특화 정보 부재 |
| 신혼 생활 | 임신/출산/육아 정보 | 맘카페, 개별 앱 | 결혼 준비와 단절 |
| 신혼 생활 | 양가 가족 행사 관리 | 수기 관리 | 체계적 관리 도구 없음 |
| 공통 | 웨딩/신혼 용품 중고거래 | 당근마켓, 번개장터 | 카테고리 특화 부재 |
| 공통 | 비슷한 처지의 커플 소통 | 맘카페, 웨딩카페 | 분산되고 익명성 부족 |

### 2.3 목적

1. 결혼 준비부터 신혼 생활까지 끊김 없는 통합 경험 제공
2. 투명한 가격 정보와 리뷰 기반의 합리적 의사결정 지원
3. 커플 단위의 공동 관리 및 소통 환경 구축
4. 신혼부부 특화 커뮤니티 및 중고거래 생태계 형성
5. B2B 업체를 위한 효과적인 고객 접점 플랫폼 제공

## 3. 시장 분석 및 벤치마킹

### 3.1 시장 규모

- 글로벌 웨딩 플래닝 앱 시장: 2024년 $12억 → 2033년 $35억 (CAGR 12.5%)
- 한국 웨딩 서비스 시장: 연 약 20조 원
- 아시아태평양 지역이 글로벌 수요의 약 38% 차지

### 3.2 경쟁사 벤치마킹

| 서비스 | 핵심 기능 | 강점 | 약점 | 벤치마킹 포인트 |
|--------|----------|------|------|----------------|
| **웨딩북** | 웨딩홀 예약, 스드메 비교, 커뮤니티 | 14만+ 리뷰, 활발한 커뮤니티 | 결혼 후 사용 중단, 신혼 기능 없음 | 커뮤니티 구조, 리뷰 시스템 |
| **오딩** | 비대면 웨딩, 스몰웨딩, 가격 투명성 | MZ세대 타겟, 합리적 가격 | 기능 범위 제한적 | 가격 투명성 UX |
| **요즘웨딩** | AI 기반 추천, 일정 관리 | 데이터 기반, 맞춤 추천 | 신규 서비스, 사용자 기반 적음 | AI 추천 엔진 |
| **아이웨딩** | 웨딩홀, 패키지, 가격 공개 | 정찰제, 앱 내 톡 | 올드한 UI, 신혼 기능 없음 | 가격 공개 모델 |
| **푸딩** | AI 플래너, 온라인 축의대 | AI 서비스, 혁신적 기능 | 아직 초기 단계 | AI 기능, 축의대 |
| **당근마켓** | 지역 기반 중고거래 | 거대 사용자 기반 | 웨딩 특화 아님 | 중고거래 UX, 지역 기반 |
| **인스타그램** | 사진 중심 SNS | 글로벌 플랫폼 | 웨딩 전용 아님 | 피드 UX, 해시태그 |

### 3.3 차별화 전략

기존 서비스들이 **결혼 준비**에만 집중하는 반면, 웨딩메이트는:

1. **라이프사이클 확장**: 결혼 준비 → 신혼 생활 → 육아 초기까지 연속적 서비스
2. **커머스 + 커뮤니티 + 콘텐츠**: 3C 전략으로 락인(Lock-in) 효과
3. **중고거래 내재화**: 웨딩/신혼 특화 중고거래로 유저 간 경제적 순환 구조
4. **지역 기반 SNS**: 같은 지역의 예비부부/신혼부부 네트워킹

## 4. 타겟 사용자 분석

### 4.1 Primary Persona

**페르소나 1: 예비신부 김하은 (29세)**
- 직업: IT기업 마케터
- 결혼 준비 단계: 예식장 선정 완료, 스드메 비교 중
- Pain Point: 업체별 가격 비교가 어렵고, 리뷰를 여러 플랫폼에서 찾아야 함
- Needs: 한 곳에서 비교/예약/리뷰를 보고 싶음, 예비신랑과 일정/예산 공유
- 디지털 리터러시: 높음

**페르소나 2: 신혼 1년차 박민수 (32세)**
- 직업: 스타트업 개발자
- 신혼 상태: 결혼 8개월차, 아내 임신 초기
- Pain Point: 양가 부모님 생신/제사 일정 관리, 태교 정보 수집, 신혼집 인테리어
- Needs: 가족 행사 리마인더, 임신/출산 가이드, 같은 처지의 부부 커뮤니티
- 디지털 리터러시: 높음

**페르소나 3: 업체 사장님 이정민 (45세)**
- 직업: 스드메 업체 대표
- Pain Point: 신규 고객 확보, 기존 플랫폼 수수료 부담
- Needs: 합리적 수수료의 입점 플랫폼, 고객 리뷰 관리 도구, 매출 분석

### 4.2 사용자 세그먼트

| 세그먼트 | 연령대 | 특징 | 예상 비중 |
|---------|--------|------|----------|
| 예비부부 (결혼 준비) | 25~35세 | 결혼 D-12개월~D-day, 활발한 검색/비교 활동 | 40% |
| 신혼부부 (결혼 0~3년) | 27~37세 | 신혼집, 임신/출산, 가족관계 관심 | 35% |
| 예비맘/대디 (육아 초기) | 28~38세 | 출산 전후, 육아 용품 필요, 커뮤니티 활동 | 15% |
| 웨딩/신혼 업체 | - | 상품/서비스 판매, 고객 접점 확보 | 10% |

## 6. 회원 체계 설계

### 6.1 회원 유형

회원 체계는 세 가지 주요 역할(Role)로 구분하되, 하나의 통합 계정 시스템 위에 역할(Role)을 부여하는 구조로 설계합니다.

| 역할 | 코드명 | 설명 | 주요 권한 |
|------|--------|------|----------|
| **관리자** | ADMIN | 플랫폼 운영 관리자 | 전체 시스템 관리, 콘텐츠 관리, 업체 승인, 통계 대시보드 |
| **일반 회원** | USER | 결혼 준비/신혼 개인 고객 | 서비스 이용, SNS 게시, 중고거래, 리뷰 작성, 상품 구매 |
| **비즈니스 회원** | BUSINESS | 상품/서비스를 판매하는 업체 | 상품 등록, 주문 관리, 리뷰 응답, 광고 집행, 매출 대시보드 |

### 6.2 회원 등급 체계

**일반 회원 (USER)** — 활동 기반 등급

| 등급 | 조건 | 혜택 |
|------|------|------|
| 🌱 새싹 | 가입 즉시 | 기본 기능 이용 |
| 🌸 예비 | 프로필 완성 + 활동 10회 | 업체 쿠폰, 중고거래 수수료 할인 |
| 💍 메이트 | 리뷰 5건 이상 + 활동 50회 | 프리미엄 콘텐츠, 우선 상담 |
| 👑 베스트 | 리뷰 20건 이상 + 추천 10회 | VIP 혜택, 업체 특별 할인, 베타 테스터 |

**비즈니스 회원 (BUSINESS)** — 인증 기반 등급

| 등급 | 조건 | 혜택 |
|------|------|------|
| 일반 입점 | 사업자 등록증 인증 | 기본 상품 등록 (월 10개) |
| 인증 업체 | 실적 검증 + 리뷰 평점 4.0+ | 상품 등록 무제한, 검색 노출 우선 |
| 프리미엄 파트너 | 월 매출 기준 충족 + 평점 4.5+ | 홈 배너 노출, 전용 매니저, 수수료 할인 |

### 6.3 인증 체계

- 일반 회원: 소셜 로그인 (카카오, 네이버, 구글, Apple) + 본인 인증 (선택)
- 비즈니스 회원: 소셜 로그인 + 사업자 등록증 인증 (필수) + 추가 서류 심사
- 커플 연동: 초대 코드 기반으로 커플 계정 연동 (일정/예산/체크리스트 공유)
- SNS 게시 권한: 일반 회원만 게시 가능 (비즈니스 회원은 광고 영역에서만 노출)

### 6.4 계정 상태

| 상태 | 설명 |
|------|------|
| ACTIVE | 정상 이용 |
| DORMANT | 90일 이상 미접속 |
| SUSPENDED | 운영 정책 위반으로 이용 정지 |
| WITHDRAWN | 탈퇴 처리 (개인정보 30일 보관 후 파기) |

## 10. SNS 기능 설계

### 10.1 설계 원칙

1. **일반 회원(USER)만 게시 가능** — 비즈니스 회원은 광고 영역에서만 콘텐츠 노출
2. **지역 기반** — 모든 게시글에 지역 태그 필수 (최소 시/도 단위)
3. **실명/반실명** — 프로필 기반 (닉네임 허용하되, 본인 인증 배지 제공)
4. **웨딩/신혼 특화** — 카테고리 해시태그로 콘텐츠 분류

### 10.2 피드 구조

**인스타그램형 카드 피드:**
- 사진/동영상 (최대 10장, 동영상 60초)
- 캡션 (최대 2,000자)
- 해시태그 (최대 30개)
- 위치 태그 (지역 필수)
- 업체/상품 태그 (선택)
- 좋아요, 댓글, 저장, 공유

**피드 알고리즘:**
- 기본: 시간순 + 관심사 가중치
- 지역 피드: 설정된 지역 기준 필터링
- 팔로잉 피드: 팔로우한 유저의 게시글만
- 인기 피드: 좋아요/댓글 기반 인기글

### 10.3 콘텐츠 가이드라인

- 결혼 준비 후기, 업체 리뷰, 신혼 생활 팁, 육아 일상 등 주제 권장

## 11. 중고거래 기능 설계

### 11.1 설계 원칙

1. **웨딩/신혼 특화 카테고리** — 일반 중고거래와 차별화
2. **안전거래 기본** — 에스크로 시스템으로 사기 방지
3. **지역 기반** — 동네 직거래 또는 택배 거래 선택
4. **상태 표준화** — 미개봉/S급/A급/B급/C급 상태 기준 통일

### 11.2 거래 프로세스

```
[판매자: 상품 등록]
        ↓
[구매자: 상품 검색/발견]
        ↓
[구매자 ↔ 판매자: 채팅으로 협의]
        ↓
  ┌─────────────┬──────────────┐
  │   직거래     │   택배거래    │
  │ (위치 약속)  │ (안전거래)    │
  └──────┬──────┴──────┬───────┘
         ↓             ↓
  [거래 완료]    [결제 → 배송 → 수령확인 → 정산]
         ↓             ↓
    [상호 평가]    [상호 평가]
```

### 11.3 안전거래 시스템

- 구매자 결제 시 웨딩메이트가 금액 보관 (에스크로)
- 판매자 발송 → 운송장 등록
- 구매자 수령 후 확인 버튼 → 판매자에게 정산
- 미확인 시 수령 후 72시간 자동 확인 처리
- 분쟁 발생 시 CS팀 중재

### 11.4 사기 방지

- 본인 인증 완료 유저만 거래 가능
- 거래 매너 온도 시스템 (당근마켓 벤치마킹)
- 의심 거래 AI 자동 탐지
- 신고 누적 시 거래 제한 및 계정 정지

### 15.2 주요 테이블 DDL (PostgreSQL)

```sql
-- Users
CREATE TABLE users (
    id              BIGSERIAL PRIMARY KEY,
    uuid            UUID NOT NULL DEFAULT gen_random_uuid() UNIQUE,
    email           VARCHAR(255) NOT NULL UNIQUE,
    nickname        VARCHAR(50) NOT NULL,
    profile_image   VARCHAR(500),
    role            VARCHAR(20) NOT NULL DEFAULT 'USER',  -- USER, BUSINESS, ADMIN
    grade           VARCHAR(20) NOT NULL DEFAULT 'SPROUT',
    status          VARCHAR(20) NOT NULL DEFAULT 'ACTIVE',
    social_provider VARCHAR(20) NOT NULL,  -- KAKAO, NAVER, GOOGLE, APPLE
    social_id       VARCHAR(255) NOT NULL,
    wedding_status  VARCHAR(20),  -- PREPARING, NEWLYWED, PARENTING
    wedding_date    DATE,
    region_code     VARCHAR(10),
    created_at      TIMESTAMP NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMP NOT NULL DEFAULT NOW()
);

-- Couples
CREATE TABLE couples (
    id              BIGSERIAL PRIMARY KEY,
    invite_code     VARCHAR(6) NOT NULL UNIQUE,
    user1_id        BIGINT NOT NULL REFERENCES users(id),
    user2_id        BIGINT REFERENCES users(id),
    status          VARCHAR(20) NOT NULL DEFAULT 'PENDING', -- PENDING, CONNECTED, DISCONNECTED
    connected_at    TIMESTAMP,
    created_at      TIMESTAMP NOT NULL DEFAULT NOW()
);

-- Posts (SNS)
CREATE TABLE posts (
    id              BIGSERIAL PRIMARY KEY,
    user_id         BIGINT NOT NULL REFERENCES users(id),
    content         TEXT,
    region_code     VARCHAR(10) NOT NULL,
    location_name   VARCHAR(200),
    latitude        DECIMAL(10,8),
    longitude       DECIMAL(11,8),
    like_count      INT NOT NULL DEFAULT 0,
    comment_count   INT NOT NULL DEFAULT 0,
    is_deleted      BOOLEAN NOT NULL DEFAULT FALSE,
    created_at      TIMESTAMP NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMP NOT NULL DEFAULT NOW()
);

-- SecondHand Items
CREATE TABLE secondhand_items (
    id              BIGSERIAL PRIMARY KEY,
    seller_id       BIGINT NOT NULL REFERENCES users(id),
    title           VARCHAR(200) NOT NULL,
    description     TEXT NOT NULL,
    category        VARCHAR(50) NOT NULL,
    condition       VARCHAR(20) NOT NULL,  -- UNOPENED, S, A, B, C
    price           BIGINT NOT NULL,
    region_code     VARCHAR(10) NOT NULL,
    trade_type      VARCHAR(20) NOT NULL,  -- DIRECT, DELIVERY, BOTH
    status          VARCHAR(20) NOT NULL DEFAULT 'ON_SALE', -- ON_SALE, RESERVED, SOLD, HIDDEN
    view_count      INT NOT NULL DEFAULT 0,
    like_count      INT NOT NULL DEFAULT 0,
    bumped_at       TIMESTAMP,
    created_at      TIMESTAMP NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMP NOT NULL DEFAULT NOW()
);
```
